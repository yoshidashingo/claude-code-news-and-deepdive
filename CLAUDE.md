# Claude Code News & Deep Dive - 記事生成ガイドライン

このファイルは Claude Code CLI が記事を自動生成する際の指示書です。

## プロジェクト概要

Claude Code のニュース・更新情報を監視し、日本語の詳細な技術解説記事を生成するシステムです。
記事は `docs/` ディレクトリに Markdown ファイルとして出力します。

## 記事テンプレート

記事は以下の frontmatter とセクション構成に従ってください:

```markdown
---
title: "（記事タイトル）"
date: YYYY-MM-DD
source: github-release | changelog | blog
source_url: "（元のURL）"
identifier: "（バージョン番号またはスラッグ）"
---

# （記事タイトル）

## 概要

（この更新・記事の要点を2-3文で簡潔にまとめる）

## 詳細

（メインコンテンツ。ソースの種類に応じて以下のセクション構成を使用）

## 技術的なポイント

（開発者にとって重要な技術的ポイントを箇条書きで整理）

## まとめ

（記事の要点を振り返り、今後の展望や影響について言及）
```

## ソース別の解説方針

### GitHub Release 記事

- リリースノートの各変更点を詳しく解説
- バグ修正は具体的にどのような問題が解決されたかを説明
- 新機能は使い方の例を含める
- Breaking Changes がある場合は移行方法を詳しく説明
- セクション構成: 概要 → 新機能 → 改善点 → バグ修正 → Breaking Changes（該当する場合）→ 技術的なポイント → まとめ

### Changelog 記事

- バージョンごとの変更点を体系的に整理
- GitHub Release と内容が重複する場合は、Changelog 独自の情報や視点を補足
- 複数バージョンにまたがるトレンドや方向性があれば言及
- セクション構成: 概要 → 変更点の詳細 → 技術的なポイント → まとめ

### Blog 記事

- ブログ記事の主要なメッセージと背景を解説
- 技術的な発表内容を開発者の視点で深掘り
- Claude Code の開発ワークフローへの影響を考察
- セクション構成: 概要 → 背景 → 発表内容の詳細 → 開発者への影響 → 技術的なポイント → まとめ

## 日本語の書き方ガイドライン

- **文体**: 「です・ます」調で統一
- **技術用語**: 英語のまま使用（例: Claude Code, GitHub Actions, API）。ただし初出時に必要であれば括弧内で日本語の説明を添える
- **コードブロック**: 言語指定付きの fenced code block を使用
- **見出し**: `##` から開始（`#` は記事タイトル用）
- **箇条書き**: `-` を使用
- **強調**: `**太字**` を使用
- **読者想定**: Claude Code を使用している日本語話者の開発者
- **トーン**: 技術ブログとして客観的かつ分かりやすく、必要に応じて実用的なアドバイスを含める

## ファイル命名規則

```
docs/YYYY-MM-DD-{source}-{identifier}.md
```

- **GitHub Release**: `docs/2026-02-07-github-release-v2.1.34.md`
- **Changelog**: `docs/2026-02-07-changelog-v2.1.34.md`
- **Blog**: `docs/2026-02-07-blog-claude-code-announcement.md`

## 注意事項

- 記事の生成時には、必ずソースURL にアクセスして最新の情報を取得してください
- 推測や憶測は避け、ソースに基づいた正確な情報を記載してください
- ソースの情報が不十分な場合は、その旨を明記してください
- 1つの記事は 1000〜3000 字程度を目安にしてください
